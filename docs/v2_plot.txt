This is not a form-building project. This is a data platform with four layers:

Layer	What It Does	What It's NOT
Input Layer	Captures applicant data with contextual UI (education path adapts to selections)	A static HTML form

Functional Requirements
FR-1: Education Path Handling (Indian System)
The current version only captures degree information. Indian education has multiple valid pathways. Your system must handle all of them.

Indian Education Pathways (Simplified)

Path A (Standard):          Path B (Diploma):         Path C (Lateral):
  10th (SSC/ICSE/CBSE)       10th                      10th
       │                        │                        │
      12th (HSC/ISC/CBSE)    Diploma (3yr)              ITI / Vocational
       │                        │                        │
    UG Degree                UG Degree                 Diploma
       │                    (Lateral Entry)               │
    PG / Work                   │                     UG Degree
                            PG / Work                    │
                                                      PG / Work
What to capture per education entry:

Field	Type	Validation Rule
Level	Dropdown	10th, 12th, Diploma, ITI, UG, PG, PhD
Board/University	Text	Required. Auto-suggest from known Indian boards/universities if possible
Stream/Specialization	Text	Required for 12th+. E.g., Science, Commerce, Arts, CSE, ECE
Year of Passing	Number	Must be ≤ current year. Must follow chronological order (12th year > 10th year)
Score	Number	Percentage or CGPA — system must ask which scale and normalize internally
Score Scale	Dropdown	Percentage / CGPA (out of 10) / CGPA (out of 4) / Grade
Backlog Count	Number	0+ for UG/PG. Hidden for 10th/12th
Gap After This Level	Number (months)	0+. Captures career gaps between education levels
Validation rules that MUST be enforced server-side:

10th is mandatory for all paths
If Path A → 12th is mandatory before UG
If Path B → Diploma is mandatory; 12th is optional
Year of passing must be chronologically increasing across levels
Score normalization: if CGPA out of 10, internally convert to percentage (×9.5 approximation or institution-specific if known). If CGPA out of 4, use standard conversion table.
Total education gap > 24 months → flag for review (not reject — flag)
Backlog > 0 at any level → flag for review
FR-2: Work Experience Module
Not captured at all in v1. This is a critical gap — most programs consider work experience in admissions.

Capture per work entry:

Field	Type	Validation
Company Name	Text	Required
Designation	Text	Required
Domain/Industry	Dropdown	IT, Non-IT, Government, Startup, Freelance, Other
Start Date	Date	Must be after last education end date (or concurrent if part-time)
End Date	Date / "Present"	If present, calculate tenure dynamically
Employment Type	Dropdown	Full-time, Part-time, Internship, Contract, Freelance
Key Skills Used	Multi-select or Tags	Free text with tag-style input
Derived fields (auto-calculated, not entered by user):

Total work experience (months) — sum of all tenures, handle overlaps
Domain-relevant experience (months) — filter by IT/relevant domains
Career gap between jobs (months) — flag if > 6 months
Current employment status (Employed / Notice Period / Unemployed)
FR-3: Validation Engine (Backend — This Is Where v1 Failed)
Your validation must happen server-side. Client-side validation is a UX convenience. Server-side validation is a data integrity requirement. These are different things.

Validation tiers:

Tier 1: HARD REJECT (data cannot be saved)
├── Missing mandatory fields (name, email, phone, 10th details)
├── Duplicate application (same email or phone already exists)
├── Chronologically impossible dates (graduated before enrolling)
├── Age < 18 at time of application
└── Score values outside valid range (e.g., percentage > 100)

Tier 2: SOFT FLAG (data saved, but flagged for manual review)
├── Education gap > 24 months
├── Active backlogs > 0
├── Work experience < 0 months (arithmetic error in dates)
├── Score below program-defined threshold
├── More than 3 career transitions across unrelated domains
└── No work experience but > 3 years since last education

Tier 3: ENRICHMENT (auto-computed metadata attached to record)
├── Applicant risk score (see Intelligence Layer)
├── Normalized scores across all education levels
├── Total experience bucket: Fresher / Junior (0-2yr) / Mid (2-5yr) / Senior (5+yr)
└── Application completeness percentage
How validation should work technically:

All form data sent as JSON payload to a backend endpoint
Backend validates against Tier 1 rules → if fail, return error response with specific field-level messages
If Tier 1 passes → run Tier 2 checks → attach flags to record
Run Tier 3 enrichment → attach computed fields
Only then write to persistent storage
⚠️ The offer letter is NOT released from this platform. IIT issues offer letters separately. This platform captures, validates, and consolidates applicant data. Get this distinction right.

FR-4: Intelligence Layer (What Makes This an AI Project)
Pick at least two from the following. You're not building ML models from scratch — you're integrating intelligence into a data pipeline.

Intelligence Feature	What It Does	Suggested Approach
Applicant Risk Scoring	Assign a 0-100 risk score based on gaps, backlogs, score trends, work relevance	Rule-based weighted scoring → later can be replaced with ML model
Anomaly Detection	Flag applications with unusual patterns (e.g., PhD holder applying for entry-level, or 5 career switches in 3 years)	Statistical thresholds on key metrics, or simple clustering
Auto-Categorization	Classify applicants into segments: "Strong Fit", "Needs Review", "Weak Fit"	Decision tree logic based on composite scores
Predictive Dropout Risk	Based on similar historical profiles, predict likelihood of dropout	If historical data available — logistic regression. Otherwise, rule-based proxy
Smart Form Assistance	Use an LLM to help users fill fields correctly (e.g., suggest university names, validate board names, explain what "lateral entry" means)	API call to Gemini/GPT for contextual help within the form
Data Quality Scoring	Score each application's data completeness and consistency	Weighted checklist algorithm
Minimum bar: Your intelligence layer must produce a quantitative output (a score, a classification, a flag) — not just a visualization. It must be computed programmatically, not manually.

FR-5: Persistent Storage & Reporting
Every validated application must be written to a shared, live-updating Google Sheet that stakeholders can view in real-time.

Implementation approach:

# Conceptual flow — you figure out the exact implementation

# Option A: Google Sheets API (recommended)
# 1. Create a Google Cloud project (free tier)
# 2. Enable Google Sheets API
# 3. Create a service account → download credentials JSON
# 4. Share your target Sheet with the service account email
# 5. Use gspread or google-api-python-client to write rows

# Option B: Google Apps Script webhook
# 1. Create a Google Sheet
# 2. Extensions → Apps Script
# 3. Write a doPost(e) function that appends data to the sheet
# 4. Deploy as web app → get URL
# 5. POST validated JSON to that URL from your backend

# Option C: Export to Excel/CSV on Google Drive
# 1. Use openpyxl or pandas to create .xlsx
# 2. Upload to Google Drive via API or manually
# 3. Script runs on schedule (cron) to update
The Google Sheet must contain:

One row per validated applicant
All captured fields + all derived/computed fields
Risk score and categorization from intelligence layer
Timestamp of submission
Validation status (Clean / Flagged / Under Review)
Sheet should be read-only for viewers — only the platform writes to it
Bonus (not required but impressive): A summary sheet or dashboard sheet that auto-computes:

Total applications received
Distribution by risk category
Average scores by education path
Applications per day (trend)

What You Will Be Evaluated On
Evaluation Rubric: 100 Points Total
Scores are assigned per dimension. Logic and outcomes carry 3× the weight of UI.

1. Validation Engine Quality — 30 points
Why this matters: Every backend engineer's first job is making sure bad data never enters the system. Interviewers at Flipkart, Razorpay, and every fintech will test this.

Score	Criteria
25-30	All Tier 1 rules enforced server-side. Tier 2 flags working. Field-level error messages. Handles edge cases (diploma path, overlapping work dates, CGPA conversion).
18-24	Most Tier 1 rules work. Some edge cases missed. Error messages exist but are generic.
10-17	Basic required-field validation only. Mix of client-side and server-side. Several rules missing.
0-9	Validation is mostly client-side required attributes. No server-side logic.
How it will be checked: Evaluator will submit 5 test applications — 2 clean, 2 with Tier 1 violations, 1 with Tier 2 flags. Your system must correctly accept, reject, or flag each one.

2. Intelligence Layer — 25 points
Why this matters: The difference between a "developer" and an "AI engineer" is the ability to embed intelligence into systems. This is literally the job you're training for.

Score	Criteria

13-19	One intelligence feature working well with clear output. Second feature partially implemented or conceptual.
6-12	Intelligence layer exists but output is cosmetic (e.g., just color-coding without actual computation behind it).
0-5	No intelligence layer. Or a feature that's just a hardcoded if-else with no real analytical value.
How it will be checked: Evaluator will ask you to explain your scoring/classification logic for a specific applicant record. "Why did this person get a risk score of 73?" You must answer with the actual computation.

What "Done" Looks Like
A stranger opens your live URL. They fill in an application as a diploma-path student with 3 years of work experience and a 6-month career gap. They submit.

The form adapts to their education path (diploma fields shown, 12th fields hidden)
Server-side validation catches any errors and returns specific field-level messages
On success, the data appears in the Google Sheet within seconds
